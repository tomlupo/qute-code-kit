# Commit Messages

Use Conventional Commits so `/ship` can correctly determine the next version
bump and group CHANGELOG entries.

## Format

```
<type>: <summary under 50 chars>

<body: what changed and why, not just what>
```

## Types and version impact

| Type | When | Bumps |
|------|------|-------|
| `feat` | New feature or capability | minor |
| `fix` | Bug fix | patch |
| `perf` | Performance improvement | patch |
| `refactor` | Code restructuring, no behavior change | none |
| `docs` | Documentation only | none |
| `chore` | Build, deps, config, tooling | none |
| `test` | Adding or fixing tests | none |
| `style` | Formatting, no code change | none |

Append `!` after the type (e.g. `feat!:`) or include a `BREAKING CHANGE:`
footer to trigger a major bump.

## Rules

- Present-tense imperative ("Add feature", not "Added feature")
- Summary under 50 characters
- Body explains **why**, not just what
- Be specific about affected components
- Never include "Generated with Claude Code" or "Co-Authored-By" lines

## Example

```
feat: Add multi-source data routing for market-datasets

Auto-routes ticker requests to best provider based on market:
- Polish equities → Stooq
- US equities → Yahoo Finance
- Crypto → Binance via CCXT
- Macro → FRED

Eliminates manual source selection for common data fetches.
```

## CHANGELOG policy

**Never hand-edit `CHANGELOG.md`.** It is generated by `/ship` (commitizen)
from the commit history. The only way to change the CHANGELOG is to write
better commit messages. This applies on every branch, including `main`.

If a project has no `CHANGELOG.md` yet, run `/ship-setup` once to bootstrap
it, then use `/ship` to cut releases.

## After committing

- **Feature branch, more work to do**: nothing else to do.
- **Feature branch, ready to merge**: remind the user to open a PR (or merge
  locally with `git merge --no-ff`). On GitHub, the release workflow will fire
  on merge to `main`. Locally, suggest `/ship` after the merge.
- **Direct commit on `main`** (local-only flow): suggest running `/ship`
  after the commit to bump version + tag + CHANGELOG in one step.
- **If the project has no `[tool.commitizen]` in `pyproject.toml`**: suggest
  running `/ship-setup` first.
