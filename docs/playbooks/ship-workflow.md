# Ship Workflow Playbook

How to use `/ship` for automated versioning, changelogs, and PRs. Covers project setup, the ship flow, and per-project conventions.

## Prerequisites

- gstack installed (`~/.claude/skills/gstack/`)
- Project has a git repo with a remote
- Feature branch workflow (main = production, feature branches for work)

## 1. One-time project setup

### Create VERSION file

Sync from pyproject.toml (Python) or package.json (Node):

```bash
# Python
grep '^version' pyproject.toml | head -1 | sed 's/version = "//;s/"//' > VERSION

# Node
node -e "console.log(require('./package.json').version)" > VERSION
```

### Initialize CHANGELOG

```bash
touch CHANGELOG.md
```

`/ship` auto-generates entries from the diff. You don't need to write the first one manually.

### Document test command in CLAUDE.md

`/ship` reads CLAUDE.md for how to run tests. Add a section:

```markdown
## Test command
uv run pytest tests/ -x
```

Or for Node projects:
```markdown
## Test command
pnpm test
```

### Verify git branch setup

```bash
# Main branch should be your production branch
git branch -m prod main  # if needed (e.g., quantlab)
git push -u origin main

# Start working on feature branches
git checkout -b feat/your-feature
```

## 2. Daily workflow

### Work on a feature branch

```bash
git checkout -b feat/add-stop-loss
# ... make changes ...
# ... commit as you go (conventional commits preferred) ...
```

### Ship when ready

```
/ship
```

That's it. `/ship` runs fully automated:

1. **Pre-flight** — checks you're on a feature branch, not main
2. **Merge main** — pulls latest main into your branch
3. **Run tests** — uses the test command from CLAUDE.md
4. **Triage failures** — distinguishes your bugs from pre-existing ones
5. **Coverage audit** — generates missing tests if needed
6. **Pre-landing review** — code review of your diff
7. **Version bump** — auto-picks PATCH, asks for MINOR/MAJOR
8. **Changelog** — auto-generates entry from the diff
9. **TODOS.md** — marks completed items, flags incomplete ones
10. **Commit** — splits into bisectable chunks
11. **Push + PR** — creates PR with full summary

### What stops the flow

- On main branch (abort — use a feature branch)
- Merge conflicts (shows conflicts, you resolve)
- Test failures caused by your changes (fix them)
- MINOR/MAJOR version decision (asks you)
- Review findings that need judgment (asks you)

### What doesn't stop the flow

- Uncommitted changes (auto-included)
- PATCH version bump (auto-decided)
- Changelog content (auto-generated)
- Commit messages (auto-written)
- Pre-existing test failures (triaged, noted in PR)

## 3. TODOS.md (optional)

A simple task tracker that lives in the repo. `/ship` reads it automatically.

```markdown
# TODOS

## In Progress
- [ ] Add stop-loss to momentum strategy

## Planned
- [ ] Backtest on 2024 data
- [ ] Add Telegram alerts for drawdown > 5%

## Done
- [x] Migrate to ccxt v4
```

At ship time, `/ship`:
- Checks if completed items in the diff match TODOS entries
- Auto-marks matching items as done
- Flags items marked "In Progress" that have no matching changes
- Asks once if you want to create TODOS.md (if it doesn't exist)

You can also use Obsidian Kanban for higher-level tracking. TODOS.md is for repo-scoped, commit-level work.

## 4. Version bump rules

`/ship` uses semver with auto-decision for PATCH:

| Change type | Version bump | Auto? |
|-------------|:------------:|:-----:|
| Bug fix, refactor, docs | PATCH (0.1.0 → 0.1.1) | yes |
| New feature, new API | MINOR (0.1.0 → 0.2.0) | asks |
| Breaking change | MAJOR (0.1.0 → 1.0.0) | asks |

VERSION file is the single source of truth. `/ship` updates it and commits.

For Python projects, keep `pyproject.toml` version in sync. `/ship` handles this if both files exist.

## 5. Changelog format

Auto-generated, written for users not contributors:

```markdown
## [0.2.0] - 2026-04-08

### Added
- Stop-loss support for momentum strategy with configurable drawdown threshold
- Telegram alerts when portfolio drawdown exceeds 5%

### Fixed
- Paper trading position sizing now matches live execution
```

## 6. Per-project conventions

### Python (quantlab, quantbox, atlas, funds-portfolios)

```
VERSION              ← semver, source of truth
CHANGELOG.md         ← auto-generated by /ship
pyproject.toml       ← version synced from VERSION
CLAUDE.md            ← must have "## Test command"
tests/               ← pytest
```

Branch workflow:
```bash
git checkout -b feat/new-strategy
# work...
/ship
# → PR created, VERSION bumped, CHANGELOG updated
# merge PR on GitHub
```

### Node/TypeScript (collaboo, mz-workouts)

```
VERSION              ← semver, source of truth
CHANGELOG.md         ← auto-generated by /ship
package.json         ← version synced from VERSION
CLAUDE.md            ← must have "## Test command"
tests/ or __tests__/ ← vitest/jest/playwright
```

Same branch workflow.

### Obsidian vaults (notebook, wiki)

Not applicable. Vaults don't have versions or releases. Use git history and 5-minute auto-commits for tracking.

## 7. Guard integration

`/ship` runs inside Claude Code, so all guards apply:

- **Destructive guard** blocks `git push --force`, `git reset --hard` even during ship
- **Lakera** screens any web content fetched during review
- **Langfuse** traces the entire ship workflow (filter by `tool:Bash` + `project:<name>`)

To ship with guards off (not recommended):
```bash
CLAUDE_SKIP_GUARDS=1 claude
# then /ship inside the session
```

## 8. Dispatcher integration

Dispatcher agents can't run `/ship` (it requires user decisions for MINOR/MAJOR bumps). Dispatcher agents work on feature branches, you `/ship` from interactive sessions.

Typical flow:
1. Send task to dispatcher via Discord: "implement stop-loss for momentum strategy"
2. Agent works on `feat/stop-loss` branch
3. Agent commits and pushes
4. You open interactive session: `cd quantlab && claude`
5. `/ship` — reviews, bumps, creates PR

## Quick reference

```bash
# Setup (once per project)
echo "0.1.0" > VERSION
touch CHANGELOG.md
# Add "## Test command" to CLAUDE.md

# Daily
git checkout -b feat/thing
# ... work ...
/ship

# That's it.
```
